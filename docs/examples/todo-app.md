# Example 01: Todo App

**Source:** [`examples/01-todo-app/`](https://github.com/bazokhan/backend-generator/tree/main/examples/01-todo-app)

The simplest possible TGraph project. One model, no auth, no dashboard — just a clean REST API with pagination and Swagger docs.

## What This Demonstrates

- `// @tg_form()` — marks a model for code generation
- `/// @tg_readonly` — fields excluded from create/update DTOs
- Flat `UserConfig` with authentication disabled
- `nonInteractive: true` for scripted (non-interactive) generation
- Pagination and search out of the box

## Prerequisites

- [Node.js](https://nodejs.org/) 18 or higher
- [npm](https://www.npmjs.com/) 9 or higher
- NestJS CLI: `npm install -g @nestjs/cli`

## Step-by-Step Setup

### 1. Navigate to the example

```bash
cd examples/01-todo-app
```

### 2. Install dependencies

```bash
npm install
```

### 3. Set up the database

```bash
npx prisma migrate dev --name init
```

This creates a local SQLite database at `prisma/dev.db` and runs Prisma Client generation.

### 4. Generate the API

```bash
npx tgraph all
```

TGraph reads `prisma/schema.prisma`, finds the `Todo` model (marked with `// @tg_form()`), and generates:

```
src/todo/
├── todo.module.ts              ← NestJS module (auto-created)
├── todo.tg.controller.ts       ← GET/POST/PUT/DELETE /api/todos
├── todo.tg.service.ts          ← CRUD service using Prisma
├── create-todo.tg.dto.ts       ← { title, description?, completed? }
└── update-todo.tg.dto.ts       ← Partial<CreateTodoDto>
```

It also generates shared infrastructure files (pagination DTOs, interceptors, decorators, utils) and updates `src/app.module.ts` to import `TodoModule`.

### 5. Start the server

```bash
npm run start:dev
```

You should see:

```
Server running at http://localhost:3001
Swagger UI at http://localhost:3001/api
```

## Verify It Works

### Open Swagger UI

Navigate to **[http://localhost:3001/api](http://localhost:3001/api)** — you'll see all five endpoints with try-it-out support.

### Create a todo

```bash
curl -X POST http://localhost:3001/api/todos \
  -H "Content-Type: application/json" \
  -d '{"title": "Buy groceries", "description": "Milk, eggs, bread"}'
```

Expected response (201):

```json
{
  "data": {
    "id": "...",
    "title": "Buy groceries",
    "description": "Milk, eggs, bread",
    "completed": false,
    "createdAt": "...",
    "updatedAt": "..."
  }
}
```

### List todos (paginated)

```bash
curl "http://localhost:3001/api/todos?page=1&limit=10"
```

Expected response (200):

```json
{
  "data": [...],
  "total": 1,
  "page": 1,
  "limit": 10
}
```

### Update a todo

```bash
curl -X PUT http://localhost:3001/api/todos/<id> \
  -H "Content-Type: application/json" \
  -d '{"completed": true}'
```

### Delete a todo

```bash
curl -X DELETE http://localhost:3001/api/todos/<id>
```

## Schema

```prisma
// @tg_form()
model Todo {
  id          String   @id @default(uuid())
  title       String
  description String?
  completed   Boolean  @default(false)
  /// @tg_readonly
  createdAt   DateTime @default(now())
  /// @tg_readonly
  updatedAt   DateTime @updatedAt
}
```

## Key Concepts

### `@tg_readonly` fields

Fields annotated with `/// @tg_readonly` are excluded from create/update DTOs. The API still returns them in responses — they're just not settable by the client.

```typescript
// Generated create-todo.tg.dto.ts
export class CreateTodoDto {
  @IsString()
  @IsNotEmpty()
  title: string;

  @IsString()
  @IsOptional()
  description?: string;

  @IsBoolean()
  @IsOptional()
  completed?: boolean;

  // createdAt and updatedAt NOT included — they're @tg_readonly
}
```

### Pagination

The generated controller uses `PaginatedSearchQueryDto` for list endpoints, supporting:

```
?page=1&limit=10&search=groceries&sortBy=createdAt&sortOrder=desc
```

### `.tg.` file convention

Generated files use a `.tg.` infix in their names (`todo.tg.controller.ts`, `create-todo.tg.dto.ts`). This marks them as safe to overwrite on re-generation. Any file without `.tg.` — like `todo.module.ts` — is only created once and never overwritten.

## Regenerating After Schema Changes

When you update your schema, just re-run:

```bash
npx tgraph api
```

Only `.tg.`-suffixed files are overwritten. Any custom code you've added to other files is preserved.
